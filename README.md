1. 깃허브 레포와 연결하기
== 테스트 진행 개인 레포를 만들고 진행해보기 ==
    1. cd ./ (폴더명)프로젝트 폴더로 진입 
    2. git init -> 깃 초기화 시켜주기
    3. 레포지토리 연결 git remote add origin (깃허브 주소)
    4. git remote -v 연결된 레포 확인하기
    5. 레포 원격 브랜치로 커밋, commit, push 진행(파일 업로드)
        git status 작업 환경 확인
        git add . 
        git commit -m "first commit" (git 커밋 메시지 규격은 따로 존재함)
        git push origin main (만약 본인의 브랜치명이 main이 아닌 master라면) -> git branch -m master main 작성
== 같성팀 레포에 작업물 올려보기 == ++간단 코드리뷰 진행
    1. 해당 프로젝트에서 자신의 기능(이름)의 브랜치를 생성한다()
    2. 프로젝트를 자신의 계정으로 fork -> main과 자신이 생성한 브랜치가 생기게 됨 (fork 할 떄 체크박스를 반드시 해제해야한다) 사실 체크하더라도 나중에 브랜치를 만들면 된다
    3. fork 한 저장소를 자신의 컴퓨터로 clone 한다 fork 한 저장소는 깃허브 홈페이지에 존재해 바로 작업이 불가능함
        자신의 작업물이 있는 저장소로 이동한다
        git clone -b (본인 아이디) --single-branch 프로젝트 주소
    ==== clone 까지 마친 상태 현재 자신의 저장소에 readme.md 파일이 존재함 ======
    4. 기능 구현을 위한 브랜치를 생성
        자신의 이름으로 가져온 브랜치에 모든 작업을 진행할 수는 없어 별도의 브랜치를 만들어 진행한다
        ex) git checkout -b LGBM_jisoo    <- 만약 자신이 해당 브랜치에서 LGBM으로 모델을 돌린다면 이런식으로도 가능하다 모델명은 다른 사람들과 동일할 수 있으니 구별하기 쉽게 작성
        git checkout -b step1  <- 기능별 이름으로 작성할 수도 있다
    ==== 현재 자신의 로컬 환경에는 자신의 이름으로 된 브랜치와 기능 브랜치 2개로 이루어져 있다 ====
    ==== 많이 헷갈리는데 지금 gigi로컬 환경의 브랜치와 원격 상태의 브랜치는 서로 다른 환경에 있다 ====
        풀어 설명하면 내가 만들고 있는 웹사이트에 누군가가 새로운 기능을 넣고싶다고 하는데 내 작업 환경을 그대로 알려준다면 사고가 일어나 fork를 통해 관리자의 허가 없이는 기능을 
        넘겨줄 수 없는 상태로 g만든것이다
        fork 를 받은 상대방은 힘들게 빌린 작업 환경을 어지럽히기 싫어 자신의 로컬환경(여긴 마구 망가져도 된다는 가정)으로 다시 끌어와 각 기능 별 구조를 만드는 것이다
    ==== 이후 작업을 진행한 후 add, commit 명령어를 진행한다(작업은 기능 브랜치 환경에서 진행) ====
        git status  // 변경된 파일 확인
        git add .   // 변경된 파일을 한번에 반영
        git commit -m "메시지"  // 작업한 내용을 메시지에 기록
    5. 본인의 원격 저장소에 올리기 로컬에서 commit 명령어를 실행하면 로컬 저장소에만 반영되고 원격에는 저장되지 않는다
        git push origin 브랜치명
        ex) git push origin step1
        push 명령어를 실행한 후의 상태는 원격 저장소에 step1 브랜치가 생성될 것이다
    ==== 여기서 부턴 코드리뷰단계 포함 ====
    6. 미션을 진행한 step1 브랜치에서 프로젝트 브랜치로 pull request를 보낸다
        브라우저에서 깃허브 저장소로 접근
        브랜치를 작업 브랜치로 변경(지금 자신의 원격 저장소)
        브랜치 우측 new pull request를 클릭
        왼쪽의 프로젝트 브랜치를 자신의 깃허브 브랜치로 변경
        팀원들 모두 리뷰어로 선정하여 깃허브 계정을 추가함
        현재 작업한 내용을 입력하고 pull request를 진행한다 + 여기서 라벨 프로젝트 마일스톤같은 디테일도 있지만 지금은 다루지 않아도 괜찮다 나중 최종 프로젝트 때 진행해도 충분해 간단 설명만
        pull request를 보냈을 때 step1 브랜치가 프로젝트의 자신의 이름이 담긴 브랜치로 작업 내용이 옮겨져 있을 것이다
    ==== 이 부분 어렵다 문서랑 책 계속 살펴봐도 어려움 ====
    7. 이제 merge가 됐다면 기존의 브랜치(step1)를 삭제하고 새로운 기능을 진행할 브랜치(step2)를 생성한다
        git checkout 본인 아이디
        git branch -D 삭제할 브랜치 이름
        ex) git checkout jisoo78
            git branch -D step1
    ==== 주의점은 해당 브랜치는 로컬에서만 삭제된거지 자신의 원격에는 존재하고 있다 ====
    8. merge한 프로젝트 저장소와 동기화하기 위해 프로젝트 저장소 추가(최초 1회 진행)
        git remote add (저장소 별칭) (저장소 url)
        ex) git remote add upstream http://sdasdsdsds.git
        git remote -v
    9. 프로젝트 저장소에서 자기 브랜치 가져오기(갱신)
        프로젝트 저장소에 fetch 명령으로 동기화 하고 싶은 저장소의 브랜치를 가져온다(자신의 로컬 환경에 merge된 작업물을 가져오는 것이다)
        git fetch upstream (본인 아이디)
        fetch 명령어 실행 후 git branch -a 실행하면 remotes/upstream/자기이름 같은 브랜치가 생성된 것을 확인할 수 있다
    10. 프로젝트 저장소와 브랜치 동기화 하기
        아직 로컬 저장소에 최신 버전의 코드가 반영된건 아니다
        rebase 명령을 실행해 프로젝트 저장소와 브랜치를 동기화 시킨다
        git rebase upstream/본인아이디
        ex)git rebase upstream/jisoo78
    ==== 이후 새로운 기능을 구현하기 위한 브랜치를 생성한다 ====
        git checkout -b step2
